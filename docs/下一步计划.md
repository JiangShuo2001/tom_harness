## 阶段一：Memory 集成

### 目标

让 Planner 的"记忆暖启动"真正生效——前面答对的题的 (task, plan) 对写入记忆，后续相似题的 Planner 检索到后可以复用成功计划结构。

### 需要的资源

| 资源 | 说明 |
|:---|:---|
| Embedding API | 阿里 DashScope `text-embedding-v3`（与 LLM 同一平台，无需额外注册），或开源 `bge-m3` 本地部署 |
| 依赖 | 若用 API：无新增（复用 `requests`）；若用本地模型：`sentence-transformers` |

### 代码修改

#### 1. 新增 embedder 适配函数

在 `tom_harness/tools/` 下新建 `embedders.py`：

```python
# tom_harness/tools/embedders.py

import requests

def dashscope_embed(text: str, api_key: str, model: str = "text-embedding-v3") -> list[float]:
    """调用 DashScope embedding API，返回归一化向量。"""
    resp = requests.post(
        "https://dashscope.aliyuncs.com/compatible-mode/v1/embeddings",
        headers={"Authorization": f"Bearer {api_key}", "Content-Type": "application/json"},
        json={"model": model, "input": text},
        timeout=30,
    )
    resp.raise_for_status()
    return resp.json()["data"][0]["embedding"]
```

构造 MemoryStore 时注入：

```python
from functools import partial
from tom_harness.tools.embedders import dashscope_embed

embedder = partial(dashscope_embed, api_key=os.environ["TOM_API_KEY"])
memory = MemoryStore(persist_path=Path("results/memory.jsonl"), embedder=embedder)
```

#### 2. 修改 benchmark 脚本 `run_tombench_harness.py`

当前问题：每条数据 `build_harness()` 新建独立的空 MemoryStore，内存不共享。

改动：
- **将 MemoryStore 提到全局**，所有 Scheduler 实例共享同一个 memory
- **打开 `persist_memories_on_success=True`**
- 传入 `persist_path` 实现 JSONL 持久化（支持断点续跑）
- MemoryStore 的 `insert()` 方法需要加线程锁（当前无锁，多线程并发写会丢数据）

```python
# 全局共享的 memory（进程级别）
import threading
_memory_lock = threading.Lock()
_shared_memory = MemoryStore(persist_path=Path("results/memory.jsonl"), embedder=embedder)

def build_harness():
    # ... 其余不变 ...
    # memory 不再新建，用全局共享
    scheduler = Scheduler(
        ...,
        memory=_shared_memory,
        config=SchedulerConfig(max_replans=1, persist_memories_on_success=True),
    )
```

#### 3. MemoryStore 加线程安全

`tools/memory.py` 的 `insert()` 和 `run()` 加 `threading.Lock`：

```python
@dataclass

    def insert(self, memory: Memory) -> None:
        with self._lock:
            # ... 现有逻辑 ...

    def run(self, *, query, **kwargs) -> dict:
        with self._lock:
            # ... 现有逻辑 ...
```

#### 4. 注册到 Registry

```python
registry.register(memory)  # tool_name="memory_retrieve" 自动暴露给 Planner
```

注册后 Planner 的 system prompt 里 `AVAILABLE TOOLS` 会包含 memory_retrieve，LLM 生成的计划就可能在步骤里调用它。

## 阶段二：Skill 集成

### 目标

为 Planner 提供结构化推理技能，让 LLM 在遇到特定类型的 ToM 题目时，按照预定义的推理流程执行，而不是自由发挥。

### 需要的资源

| 资源 | 说明 |
|:---|:---|
| 技能文件 | 11 个 `.md` 文件（声明式技能），放在 `tom_harness/skills/` 目录 |
| 无外部依赖 | 技能执行走 LLM 调用，代码层面已全部就绪 |

### 技能清单

根据 `failure_handlers.py` 中 `FAILURE_TO_SKILLS` 的映射，需要编写以下技能：

| 技能 ID | 名称 | 目标 ToM 任务 | 优先级 |
|:---|:---|:---|:---|
| S01_knowledge_state_tracking | 知识状态追踪 | 追踪"谁知道什么" | 高 |
| S02_knowledge_gate_filter | 知识门过滤 | 判断角色是否有信息获取渠道 | 高 |
| S03_false_belief_tracking | 错误信念追踪 | 一阶错误信念推理 | 高 |
| S04_second_order_belief | 二阶信念推理 | "A认为B认为..." | 中 |
| S05_double_bluff_reasoning | 双重欺骗推理 | 说谎/反讽意图识别 | 中 |
| S06_pragmatic_speech_act | 语用推理 | 言外之意、暗示 | 高 |
| S07_audience_adaptation | 听众适应 | 不同听众理解差异 | 低 |
| S08_quantifier_grounding | 量化锚定 | "几乎所有"/"大多数"的推理 | 高 |
| S09_emotion_behind_mask | 隐藏情绪 | 表面 vs 真实情感 | 中 |
| S10_perspective_integration | 多视角整合 | 整合多角色视角 | 中 |
| S11_desire_belief_interaction | 愿望-信念交互 | 行为由愿望+信念共同驱动 | 中 |

### 技能文件格式

每个 `.md` 文件遵循 frontmatter + workflow 结构，SkillLib 已支持解析：

```markdown
---
skill_id: S06_pragmatic_speech_act
name: Pragmatic Speech Act Analysis
description: Analyze indirect speech acts, hints, and implied meanings in dialogue.
triggers:
  - "hinting"
  - "implied"
  - "indirect"
  - "pragmatic"
---

## Workflow

1. **Identify the literal statement**: Extract what was explicitly said.
2. **Analyze context**: What is the speaker's situation, goal, and relationship with the listener?
3. **Infer the implied meaning**: What does the speaker actually want the listener to understand or do?
4. **Check for mismatch**: Does the literal meaning differ from the implied meaning? If so, why?
5. **Evaluate listener's response**: Does the listener understand the implied meaning?

## Output Format

Return a JSON object:
{
  "literal_meaning": "...",
  "implied_meaning": "...",
  "speaker_goal": "...",
  "mismatch": true/false,
  "listener_understanding": "..."
}
```

## 阶段三：RAG 集成

### 目标

为 Executor 提供社会规范、文化习俗等领域知识的检索能力，解决模型"缺背景知识"导致的错误。

### 需要的资源

| 资源 | 说明 |
|:---|:---|
| 社会规范知识库 | JSONL 格式，每条 `{"id": "...", "text": "...", "metadata": {"domain": "moral|conventional|cultural"}}` |
| Embedding API | 与 Memory 共用同一个 embedder |

### 知识库来源方案

| 方案 | 优缺点 |
|:---|:---|
| **A. LLM 生成** | 用 GPT-4/Claude 针对 ToMBench 题目类型批量生成社会规范条目。快速但可能有幻觉。 |
| **B. 学术数据集** | Social Chemistry 101 (Emelin et al.)、ETHICS (Hendrycks et al.)。质量高但需要格式转换和筛选。 |
| **C. 从 ToMBench 本身提取** | 分析错误案例，提取需要的背景知识，针对性构建。精准但覆盖面窄。 |

建议先用 **方案 A+C 结合**：从错误案例中提取知识缺口 → 用 LLM 针对性生成 → 人工审核。

## 每个阶段完成后的 benchmark 脚本变化

最终的 `build_harness()` 大致会变成：

```python
def build_harness(shared_memory):
    llm = LLMClient(...)
    embedder = partial(dashscope_embed, api_key=api_key)

    # 工具
    skill_lib = SkillLib(skills_dir=Path("tom_harness/skills"))
    rag = RAGEngine(corpus_path=Path("tom_harness/knowledge/social_norms.jsonl"), embedder=embedder)

    # 注册到 Registry
    registry = ToolRegistry()
    registry.register(shared_memory)     # memory_retrieve
    registry.register(skill_lib)         # execute_skill
    registry.register(rag)               # rag_retrieve

    # 上下文
    ctx = ContextManager()
    ctx.install_fixed(
        system_identity="A ToM-focused reasoning agent...",
        tool_schema_summary=registry.schema_summary(),  # 现在会列出 3 个工具
        safety_policy="...",
    )
    ctx.attach_skill_defs(skill_lib.list_skills())

    # 插件
    hooks = HookRegistry()
    from tom_harness.plugins.tom import install
    install(hooks=hooks, skill_lib=skill_lib)

    # 组装
    planner = Planner(llm=llm, registry=registry, context=ctx, hooks=hooks, memory=shared_memory)
    executor = Executor(llm=llm, registry=registry, context=ctx, hooks=hooks, skill_lib=skill_lib)
    scheduler = Scheduler(
        planner=planner, executor=executor, registry=registry, context=ctx,
        hooks=hooks, memory=shared_memory,
        config=SchedulerConfig(max_replans=2, persist_memories_on_success=True),
    )
    return scheduler
```

## 风险与注意事项

| 风险 | 影响 | 应对 |
|:---|:---|:---|
| Embedding API 调用增加成本和延迟 | 每条数据多 1-3 次 embedding 调用 | 本地缓存已 embed 过的 query；或用本地模型 |
| Memory 累积噪声（错误计划也被记住） | Planner 参考了错误计划导致准确率下降 | 只存 `success=True` 的（当前已如此）；加 score 衰减 |
| Skill 过于刚性导致灵活性下降 | 简单题被迫走复杂流程 | Skill 的 trigger 做好匹配；Planner 有选择不调用的自由 |
| RAG 检索到无关知识干扰推理 | 引入噪声 | 设置 `similarity_threshold`；知识库质量把控 |
| 多线程并发写 Memory 的一致性 | 偶发数据丢失或重复 | 加 threading.Lock（阶段一就做） |

## 2026-04-26 Skill & RAG 集成方案

### 背景

目前 `skills/`（旧版15个）和 `skill_v2/`（新版12个）两套 Skill 体系，以及 `ToMRAG` 常识检索模块，
都未能真正融入 tom_harness 的 Plan-then-Execute 架构。核心问题：

1. **Planner 不触发工具**：Planner 看到了 `rag_retrieve` 和 `execute_skill`，但选择不用（tool=none）。
2. **Skill 信息暴露不足**：Planner 只知道"有个工具叫 execute_skill"，不知道有哪些具体 skill、各自适用什么场景。
3. **RAG 数据粒度过碎**：ATOMIC/SocialChem/NormBank 存的是符号化三元组（"PersonX is hurried"），
   大模型自身参数知识已经覆盖这类常识，检索收益极低。
4. **skill_v2 是端到端解题模版**：直接命令模型输出 A/B/C/D，与 Plan-then-Execute 架构冲突。

### 方案 A：硬编码注入（快速验证，不改框架核心）

**目标**：用最小改动验证 Skill 和 RAG 的内容是否能提升准确率。

#### A1. Skill 硬编码注入

- **路由阶段**：在 `Scheduler.run()` 里，拿到题目后，先用一次额外的 LLM 调用做路由。
  把 `ROUTING.md`（或 `skill_v2/ROUTING_GUIDE.md`）+ 精简版 Skill Catalog 喂给模型，
  让它输出 `Skill: <ID>`。
- **注入阶段**：拿到选中的 skill_id 后，读取对应的完整 Skill 内容（旧版 `SKILL.md` 或新版 `skills.py` 中的字符串），
  作为额外上下文拼接到 Planner 或 Executor 的提示词中。
- **执行阶段**：Planner/Executor 在已有提示词基础上多了一段 `=== STRATEGY ===` 块，
  框架的 Plan-then-Execute 流程照常运行，但模型的推理会受到 Skill 内容的引导。
- **实现位置**：修改 `tom_harness/scheduler.py`，在 `run()` 方法开头加路由调用；
  修改 `tom_harness/context.py`，新增 `install_skill()` 方法（类似 `install_playbook()`）。

#### A2. RAG 硬编码注入

- **自动检索**：在 `Scheduler.run()` 里，拿到题目后，直接用题目文本调一次 `RAGEngine.run(query=question)`，
  取 top_k=3 结果。
- **注入阶段**：把检索到的常识段落格式化后，拼接到 Planner 的 Context 或 Executor 的 Context 中
  （类似 MemoryPlaybook 的注入方式）。
- **实现位置**：修改 `tom_harness/scheduler.py`，在 `run()` 方法里加检索调用；
  修改 `tom_harness/context.py`，新增 `install_rag_context()` 方法。

#### A 方案的验证指标

- 对比 baseline（无 skill 无 rag）、+skill、+rag、+skill+rag 四种配置的准确率。
- 重点关注 Planner 生成的 plan 质量是否因 skill 注入而改变。

### 方案 B：框架适配（符合 Agent 架构，长期方案）

**目标**：让 Skill 和 RAG 真正作为 Tool 被 Planner 主动调度。

#### B1. Skill 作为 Tool 适配

- **统一 Skill 格式**：
  - 旧版 `skills/`：修改 `SkillLib.load_dir()` 支持递归扫描子目录（`skills/skill1/SKILL.md`）。
  - 新版 `skill_v2/`：将 `skills.py` 中的 Python 字符串改写为 `.md` 文件，
    去掉"直接选 A/B/C/D"的指令，改为输出结构化中间结论。
- **Planner 路由集成**：
  - 在 Planner 的 Context 中注入 Skill Catalog（精简版：每个 skill 的 Use When + Decision Variable + 输入参数要求）。
  - 修改 `PLANNER_SYSTEM` 提示词，加入规则：
    "当某个 Skill 匹配当前任务时，规划一个专门的 Step 来委托给该 Skill，不要自己拆解 Skill 的内部逻辑。"
  - Planner 输出的 Step 中 tool_type=skill, tool_params 包含 skill_id 和 input_context。
- **Executor 执行**：
  - 现有 `SkillLib.run()` 的 declarative 路径已经可用（读取 SKILL.md body + input_context，发起独立 LLM 调用）。
  - 确保 Skill 内部的 LLM 调用输出的是中间分析结论（不是最终选项），Finalizer 再综合选答案。

#### B2. RAG 作为 Skill 的内部工具

- **从 ToolRegistry 撤下 RAG**：不再让 Planner 直接看到 `rag_retrieve`。
- **绑定到特定 Skill**：在需要社会规范知识的 Skill（如 faux-pas 检测、说服策略）的 Workflow 中，
  加入"调用 RAG 检索相关社会规范"的步骤。
  把检索结果拼入发给 LLM 的 prompt 中。

#### B 方案的验证指标

- Planner 是否能正确路由到合适的 Skill（routing accuracy）。
- Skill 执行后的中间结论质量（是否包含了关键推理信息）。
- 端到端准确率对比。
- LLM 调用次数和成本对比（B 方案会多出路由 + Skill 执行的 LLM 调用）。

## 2026-04-27 Skills 融合方案（旧版 15 个 + 新版 12 个）

### 两套 Skill 的对比

| 维度 | 旧版 `skills/` (15个) | 新版 `skill_v2/` (12个) |
|---|---|---|
| 格式 | `.md` frontmatter + Workflow + Trigger Checklist | Python 字符串常量 |
| 粒度 | **细粒度**：同一现象拆成多个子技能（faux-pas → skill1+skill2；false-belief → skill3+skill4+skill5；scalar → skill10+skill11） | **粗粒度**：按错误模式聚类，一个 skill 覆盖多个 task_type（S3 覆盖全部 belief；S2 覆盖所有 scalar） |
| 内容深度 | **框架性**：列出 workflow 步骤，但缺少具体数值规则和 guardrail | **实战性**：包含具体数值区间（"almost no = 1-3"）、worked example、guardrail、易混淆区分 |
| 路由 | `ROUTING.md`：决策变量 + 最近邻混淆 + 手工路由规则表 | `ROUTER_CATALOG` + LLM 路由 + 解析器 |
| 覆盖范围 | 仅 ToMBench | ToMBench + CogToM |

### 融合策略：以 skill_v2 为主体

**原因**：
1. skill_v2 的 prompt 内容经过多轮 error-driven 迭代，包含 worked example 和 guardrail，质量更高
2. skill_v2 的 12 个 skill 已覆盖所有旧版 15 个 skill 的场景
3. 旧版的细粒度分流（如 skill1 vs skill2）在 skill_v2 内部已通过 STEP 分支处理

### 映射关系

| skill_v2 | 对应旧版 | 融合动作 |
|---|---|---|
| S1_FauxPas | skill1 + skill2 | 直接用 S1（已包含 speaker-knowledge-gap 检查） |
| S2_Scalar | skill10 + skill11 | 直接用 S2（已包含 before/after counting 分支） |
| S3_BeliefLedger | skill3 + skill4 + skill5 | 直接用 S3（已包含 1st/2nd order + content belief） |
| S4_Strategic | skill7 + skill12 + skill13 + skill14 + skill15 | 直接用 S4（已包含 persuasion rule + intention attribution） |
| S5_Expectation | skill8 + skill9 + skill6 | 直接用 S5（已包含 hidden emotion + delta analysis） |
| S6_Spatial | 旧版无对应 | 直接用 S6 |
| S7_KnowledgeBound | 旧版无对应 | 直接用 S7 |
| S8-S12 | 旧版无对应 | 直接用 S8-S12 |

### 融合后的路由设计

使用 skill_v2 的 `ROUTER_CATALOG`（LLM 路由），从旧版 `ROUTING.md` 吸收两个改进：
1. **决策变量思维链**：路由 prompt 中加入"先识别 asked output，再匹配 decision variable"
2. **高混淆边界检查**：旧版的 boundary check 补充到 ROUTER_CATALOG 相关条目

**结论**：融合后仍然 12 个 skill，代码实体来自 `skill_v2/skills.py`，路由来自 `skill_v2/llm_router.py`。
